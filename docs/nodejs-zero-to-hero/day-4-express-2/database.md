## Database คืออะไร

คำว่า **Database (ฐานข้อมูล)** โดยทั่วไปหมายถึง “ชุดข้อมูลที่ถูกจัดระเบียบ (organized data) เพื่อให้ค้นหา แก้ไข และใช้งานร่วมกันได้อย่างมีประสิทธิภาพ”

แต่ในงานซอฟต์แวร์ เรามักพูดถึง “ระบบฐานข้อมูล” (database system) ซึ่งประกอบด้วย:

- **Data**: ข้อมูลที่เก็บ (เช่น หนังสือ, ผู้ใช้, การยืม–คืน)
- **Schema**: โครงสร้าง/กติกาของข้อมูล (เช่น ตาราง, ชนิดข้อมูล, ข้อบังคับ)
- **DBMS (Database Management System)**: ซอฟต์แวร์ที่จัดการการเก็บ/ค้น/แก้ข้อมูลให้เรา (เช่น PostgreSQL, MySQL)
- **Applications/Users**: โปรแกรมที่ใช้งาน DBMS (เช่น Express API, DBeaver)

เปรียบเทียบ:

- “ไฟล์ Excel/CSV” คือข้อมูลที่จัดเก็บได้ แต่ขาดความสามารถเชิงระบบ เช่น ควบคุมการเขียนพร้อมกัน, กู้คืนเมื่อพัง, บังคับกติกาข้อมูล
- “DBMS” คือเครื่องมือที่ทำให้ข้อมูลเป็นทรัพยากรที่แชร์ร่วมกันได้อย่างปลอดภัย และรองรับการใช้งานจริงในระดับระบบ

## ทำไมต้องใช้ DBMS (แทนที่จะเก็บเป็นไฟล์)

เหตุผลหลักที่ระบบจริงเลือกใช้ DBMS มีอย่างน้อย 6 ประเด็นสำคัญ:

1) **Concurrency (รองรับการใช้งานพร้อมกัน)**  
หลายผู้ใช้/หลายโปรแกรมเข้าถึงข้อมูลพร้อมกันได้ โดยลดปัญหา “เขียนทับ” หรือ “อ่านค่ากลางทาง”

2) **Data Integrity (ความถูกต้องของข้อมูล)**  
DBMS ช่วย “บังคับกติกา” เช่น ห้ามซ้ำ, ห้ามว่าง, ต้องอ้างอิงกันถูกต้อง (ผ่าน constraints)

1) **Transactions**  
ทำหลายคำสั่งให้สำเร็จพร้อมกัน หรือถ้าพลาดให้ยกเลิกทั้งหมด (atomicity)

1) **Query Processing & Optimization (ค้นข้อมูลอย่างมีประสิทธิภาพ)**  
DBMS มีตัวประมวลผลคำสั่งและ optimizer เพื่อให้ query ทำงานเร็วขึ้น โดยอาศัย index และสถิติ

1) **Security & Access Control (ความปลอดภัยและสิทธิ์)**  
กำหนดผู้ใช้/บทบาท (roles) และสิทธิ์การเข้าถึงตาราง/คอลัมน์ได้

1) **Backup/Recovery (สำรองและกู้คืนได้)**  
ช่วยลดความเสียหายเมื่อระบบ/เครื่องล่ม และสนับสนุนการสำรองข้อมูลอย่างเป็นระบบ

## ฐานข้อมูลเชิงสัมพันธ์ (Relational Database) แบบย่อ

ในคอร์สนี้เราจะใช้ฐานข้อมูลเชิงสัมพันธ์เป็นหลัก แนวคิดสำคัญคือ:

- ข้อมูลถูกเก็บใน **ตาราง (table / relation)**
- แถว (row) แทน **รายการ** หนึ่งรายการ เช่น หนังสือ 1 เล่ม, ผู้ใช้ 1 คน
- คอลัมน์ (column) แทน **คุณลักษณะ** เช่น ชื่อหนังสือ, ปีพิมพ์
- ตารางต่าง ๆ เชื่อมกันด้วย **คีย์ (keys)** และ **ความสัมพันธ์ (relationships)**

แนวคิดที่ต้องรู้:

- **Primary Key (PK):** ตัวระบุที่ไม่ซ้ำของแถว เช่น `book_id`
- **Foreign Key (FK):** คอลัมน์ที่อ้างถึง PK ของอีกตาราง เพื่อบังคับความสัมพันธ์ให้ถูกต้อง เช่น `borrow.user_id` อ้างถึง `users.user_id`
- **Constraints:** กติกาเพื่อคุมความถูกต้อง เช่น `NOT NULL`, `UNIQUE`, `CHECK`, `FOREIGN KEY`

ตัวอย่าง:

- `books(book_id, title, isbn, ...)`
- `users(user_id, name, ...)`
- `borrows(borrow_id, user_id, book_id, borrowed_at, due_date, returned_at, ...)`

ข้อดีของ relational model คือ “โครงสร้างชัด” และ “ตรวจสอบความถูกต้องได้” เหมาะกับข้อมูลที่มีความสัมพันธ์และต้องการความน่าเชื่อถือสูง

## SQL คืออะไร

**SQL (Structured Query Language)** คือภาษามาตรฐานสำหรับจัดการฐานข้อมูลเชิงสัมพันธ์ โดยแบ่งงานที่พบบ่อยเป็น:

- **DDL (Data Definition Language):** สร้าง/แก้โครงสร้าง เช่น `CREATE TABLE`, `ALTER TABLE`
- **DML (Data Manipulation Language):** เพิ่ม/แก้/ลบ/ดึงข้อมูล เช่น `INSERT`, `UPDATE`, `DELETE`, `SELECT`
- **DCL (Data Control Language):** จัดการสิทธิ์ เช่น `GRANT`, `REVOKE`

ตัวอย่าง query:

```sql
-- ค้นหาหนังสือที่ชื่อมีคำว่า "Node"
SELECT book_id, title
FROM books
WHERE title ILIKE '%node%';
```

```sql
-- ดูรายการยืมที่ยังไม่คืน
SELECT borrow_id, user_id, book_id, due_date
FROM borrows
WHERE returned_at IS NULL;
```

## Transaction และ ACID

**Transaction** คือหน่วยการทำงานที่ DBMS มองว่า “ต้องสำเร็จพร้อมกัน” หรือ “ยกเลิกพร้อมกัน”

คุณสมบัติ ACID ที่มักใช้เป็นกรอบอธิบาย:

- **Atomicity:** ทำทั้งหมดหรือไม่ทำเลย (all-or-nothing)
- **Consistency:** ข้อมูลที่เกี่ยวข้องกันจะต้องถูกแสดงผลออกมาอย่างถูกต้องและสอดคล้องกันเสมอ (constraints, rules)
- **Isolation:** Transaction ที่เกิดขึ้นจะต้อง independence ต่อกันไม่สามารถมีผลต่อกัน ณ ช่วงเวลาหนึ่ง (ระดับ isolation มีหลายแบบ)
- **Durability:** commit แล้วต้องไม่หาย แม้เครื่องดับ (กลไก recovery)

ตัวอย่างเชิงระบบ: การ “ยืมหนังสือ” อาจต้องทำหลายอย่าง เช่น ลดจำนวนคงเหลือ + สร้างรายการยืม ถ้าขั้นตอนใดพลาด ควรยกเลิกทั้งหมดเพื่อไม่ให้ข้อมูลเพี้ยน
